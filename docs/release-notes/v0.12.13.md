# v0.12.13 release notes

This file carries per-release notes that do not belong in `CHANGELOG.md`. The CHANGELOG is the authoritative record of what changed; this file captures honest gate outcomes and distribution-state summaries that evolve across the release window.

## External proof loop gate

At v0.12.13 tag time, the external proof loop honest gate (operating-model Rule 6) is **RED - honest miss**. No non-Originary external integration is captured as a verifiable case study under `docs/case-studies/`. This status is recorded here and carried into the next release window per the rule.

### What was attempted

- Distribution preparation on eight marketplace and IDE listing surfaces (see [`docs/case-studies/distribution-submissions.md`](../case-studies/distribution-submissions.md)). Every row is at `prepared` state with committed manifest artifacts under `packages/mcp-server/smithery.yaml` and `surfaces/plugin-pack/`.
- Admissibility rules for external proof are now documented in [`docs/case-studies/README.md`](../case-studies/README.md). A case-study template lives at [`docs/case-studies/TEMPLATE.md`](../case-studies/TEMPLATE.md).

### Why the case-study artifact did not land in v0.12.13

- No external party has executed an integration end to end against the current stable receipt format with a public, verifiable artifact that meets the admissibility checklist.
- Tag-time requirement is `prepared` or `submitted` with reproducible artifact reference on the distribution rows; that is satisfied. External case-study evidence is distinct and remains open.

### Closing target

- **Next closing target:** v0.13.0 release window.
- The external proof loop gate is carried forward explicitly, not silently downgraded.

### How to promote this gate to GREEN in a future window

- Land at least one case study under `docs/case-studies/{party}.md` that satisfies every admissibility item in [`docs/case-studies/README.md`](../case-studies/README.md).
- Record the party, artifact URL, and capture date in the corresponding release-notes file under `docs/release-notes/`.

## Distribution submissions

Distribution submission rows are tracked in [`docs/case-studies/distribution-submissions.md`](../case-studies/distribution-submissions.md). At tag time, every row is at `prepared` or `submitted` state with a reproducible artifact reference. `discoverable` is a post-release window KPI on a 30 / 60 / 90-day cadence and is not tag-blocking.

## Release mode and automation

v0.12.13 ships via Mode 2 (`next` soak followed by `latest` promotion). Two-stage GitHub Release automation runs the stage 1 draft / prerelease at `next` publish and finalizes at `latest` promotion. A post-release truth reconciler verifies npm dist-tags, the git tag, the GitHub Release state, `docs/releases/facts.json`, `docs/releases/current.json`, and `REPO_SURFACE_STATUS.json`.
